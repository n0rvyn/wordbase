## Decisions

### [DP-3.5-1] Screenshot templateUrl resolution size + format
ASC returns `{w}x{h}bb.{f}` templates (not final URLs like iTunes). Must substitute.
- **Chosen:** width `1290`, height `2796` (iPhone 6.9" portrait), format `png`. Stored as fully-resolved URLs alongside iTunes-style URLs so the frontend treats both identically. Rationale: 1290×2796 is current App Store portrait spec; `png` is lossless and ASC-supported. Single pinned size keeps the strip uniform.

### [DP-3.5-2] Category source when ASC + iTunes disagree
ASC `primaryCategory` is an `appCategories` resource whose `id` is an uppercase token (`PRODUCTIVITY`) with **no display `name` attribute** (live-verified: my `/appInfos?include=primaryCategory` call returned the category only via `id`, not `attributes.name`). The existing extractor (`asc.service.ts:164-167`) reads **only** `attributes.name`, so for the real API it yields `category=null`. iTunes gives the clean display string `Productivity`.
- **Chosen:** Keep the extractor reading display `name` only (do NOT inject the uppercase id token — that would shadow iTunes' clean `Productivity` with an ugly `PRODUCTIVITY` under the ASC-first merge, hurting display). Net real-path behavior: ASC category = `null` → merge falls through to iTunes `Productivity`. So **category is sourced from iTunes by design**, ASC's real contribution is subtitle/whatsNew/version/screenshots. Task 1 acceptance and Task 4 fixture are written to this truth (category=null out of `fetchAppMetadata`).

### [DP-3.5-3] accentColor source
Repo `AccentColor.colorset` is Git LFS (API returns pointer); reading it needs `git lfs pull`. ASC has no color.
- **Chosen:** compute dominant color from the iTunes icon art (`artworkUrl512`) at seed time; present the hex to the user for confirmation before storing. (color-from-app-art per design system, no repo dependency.)

### [DP-3.5-4] features content source
No structured icon+title+blurb source exists (ASC/iTunes only have prose description; repo `features.md` is raw ideation).
- **Chosen:** draft 3–5 `{icon,title,blurb}` from the synced Chinese description; user edits/confirms before store. MCP-managed editing of features is deferred to the reverse-integration track.

**Steps:**

2. Keep the existing `appStoreVersions` call for version/whatsNew/description unchanged (it is API-valid). **But** capture the chosen `appStoreVersionLocalization` id deterministically (prefer `locale==='zh-Hans'`, else first) — Task 2 reuses this exact id for screenshots so subtitle-locale and screenshot-locale stay aligned (see Task 2 MR-1).
3. Keep `ascFetch` throwing on non-200 (the `.catch` in app-sync still degrades gracefully — but Task 3 adds a logged warning so the swallow is observable).

## Task 2 — Add real ASC screenshot fetching

**Files:** `packages/api/src/services/asc.service.ts`

**MR-1 (verifier must-revise):** screenshots are **per version-localization**. Do NOT fetch against whatever loc the version-localization loop happened to `break` on (`asc.service.ts:202-208` breaks on first loc with attributes — locale order not guaranteed → non-deterministic screenshot count, contradicts multi-app safety). 5. If the screenshot-sets call fails or returns none, return `[]` for screenshots (do not throw the whole metadata fetch — wrap just this sub-call in try/catch returning `[]`). Empty `[]` is then handled by Task 3's merge harden (falls through to iTunes, won't shadow).

## Task 3 — Harden screenshot merge (empty ASC must not shadow iTunes)

**Files:** `packages/api/src/services/app-sync.service.ts`

**Steps:**
1. Change line 48 from `const rawShots = asc?.screenshots ?? itunes?.screenshots;` to prefer ASC only when non-empty:
   `const rawShots = (asc?.screenshots && asc.screenshots.length > 0) ? asc.screenshots : itunes?.screenshots;`
2. Leave the line-49 `length > 0 ? JSON.stringify : cur.screenshots` fallback as-is (still correct: when both sources empty, keep current DB value).
3. **Rec-3 (verifier, 禁止静默降级):** the root cause this bug survived is the silent swallow at `app-sync.service.ts:44`. Change `.catch(() => null)` to log before degrading:
   `.catch((e) => { console.warn('[app-sync] ASC fetchAppMetadata failed:', e instanceof Error ? e.message : e); return null; })`

**Verification:**
