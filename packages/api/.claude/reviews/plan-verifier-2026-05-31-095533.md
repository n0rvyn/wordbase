## Check 4 — Idempotency: null appStoreId edge case

**Evidence:**

Task 7 Step 1 maps: `name: d.attributes?.name ?? null`

`CreateAppData.name` is typed `string` (not `string | null`) at `app.service.ts:72`.

`schema.ts:186`: `name: text('name').notNull()` — DB column is NOT NULL.

If `d.attributes?.name` is absent/undefined from ASC (unlikely but possible for a partially-provisioned app), `discoverApps` will pass `name: null` into `createApp`. At runtime:
- TypeScript will not catch this (the `?? null` expression yields `null` typed as `string | null`, but the interface says `string` — TS would flag this as a type error during compilation).
- If somehow compiled with loose checks, the DB INSERT would throw a NOT NULL constraint violation.

**This is a real defect.** The plan does not specify a fallback for null name (e.g., `name: d.attributes?.name ?? d.id`). In practice ASC always returns `attributes.name` for registered apps, but the plan should either:
(a) use `name: d.attributes?.name ?? d.id` (fallback to appStoreId as name), or
(b) skip apps with null name with a `console.warn`.

Additionally: the lookup `eq(apps.appStoreId, x)` where `x = d.id` (always a string in ASC JSON:API) is safe. No null appStoreId risk on the read side.

**Verdict for Check 4:** ❌ **must-revise** — null name from ASC is a type error (`CreateAppData.name: string` is non-nullable) and would cause a DB NOT NULL constraint violation. Add fallback: `name: d.attributes?.name ?? d.id`.
